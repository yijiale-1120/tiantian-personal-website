# 项目讲解 — My Personal Website

> 本文档面向希望深入理解此项目每一层设计的读者，从全局鸟瞰到单行代码的作用逐步展开。

---

## 一、项目总览

这是一个 **前后端分离** 的个人网站，具备以下四大功能页面：

| 页面 | 路径 | 说明 |
|------|------|------|
| 个人主页 | `/` | 展示个人信息、项目经验、技术栈、时间线 |
| 登录页 | `/login` | 管理员使用账号密码获取 JWT Token |
| 反馈提交页 | `/feedback` | 登录后可提交反馈（未登录会弹窗引导登录） |
| 管理端反馈列表 | `/admin/feedbacks` | 登录后查看所有用户反馈，3 秒自动刷新 |

### 技术栈一览

```
┌─────────────── 前端 ───────────────┐    ┌─────────────── 后端 ───────────────┐
│  Next.js 16 (App Router)           │    │  NestJS 11                         │
│  React 19 + TypeScript             │    │  TypeORM + PostgreSQL              │
│  Tailwind CSS v4                   │    │  Passport JWT + bcrypt             │
│  axios + react-hook-form           │    │  class-validator                   │
│  dayjs + react-icons               │    │                                    │
└────────────────────────────────────┘    └────────────────────────────────────┘
         │  HTTP (JSON)  │
         └───────────────┘
```

### 部署方式

- **前端**：`next build` 静态导出到 `out/`，可部署到 GitHub Pages 或宝塔静态站点
- **后端**：`nest build` 编译到 `dist/`，用 PM2 管理 Node 进程运行在服务器上
- 两条 GitHub Actions 工作流自动化完成上述流程

## 三、Next.js 核心配置解读

### `next.config.ts`

```typescript
const nextConfig: NextConfig = {
  basePath: "/tiantian-personal-website",
  output: "export",
  trailingSlash: true,
  images: {
    unoptimized: true,
    remotePatterns: [{ protocol: "https", hostname: "images.unsplash.com" }],
  },
};
```

逐项解释：

| 配置项 | 值 | 作用 |
|--------|----|------|
| `basePath` | `"/tiantian-personal-website"` | 所有路由前面加上这个前缀。访问首页不再是 `/` 而是 `/tiantian-personal-website/`。这是为了在子路径下部署（如 GitHub Pages 的 `username.github.io/tiantian-personal-website`） |
| `output` | `"export"` | 构建时生成纯静态 HTML 文件到 `out/` 目录，不需要 Node.js 服务器运行前端。适合部署到任何静态托管平台 |
| `trailingSlash` | `true` | 路由末尾自动加 `/`。`/login` 变成 `/login/`，生成的静态文件是 `login/index.html`，确保静态服务器能正确路由 |
| `images.unoptimized` | `true` | 禁用 Next.js 内置的图片优化服务（因为静态导出模式下没有 Node 服务器来处理图片优化） |
| `images.remotePatterns` | `[unsplash]` | 允许加载 Unsplash 的远程图片（用于首页大图） |

### `tsconfig.json`（前端）

关键配置：

- `"paths": { "@/*": ["./src/*"] }` — 路径别名，`@/views/Home` 等价于 `./src/views/Home`
- `"exclude": ["node_modules", "backend"]` — TypeScript 编译排除后端代码，前后端互不干扰
- `"jsx": "react-jsx"` — 使用 React 17+ 的自动 JSX 转换，无需手动 `import React`

### `postcss.config.mjs`

仅注册了 `@tailwindcss/postcss` 插件。Tailwind CSS v4 不再需要独立的 `tailwind.config.js`，主题配置直接写在 `globals.css` 的 `@theme` 指令中：

```css
@theme {
  --color-primary: #0ea5e9;       /* 天蓝色主色调 */
  --color-primary-dark: #0284c7;  /* 深蓝变体 */
  --color-secondary: #8b5cf6;     /* 紫色辅助色 */
  --font-sans: "Inter", system-ui, sans-serif;
}
```

这些 CSS 变量可以直接在 Tailwind 类名中使用，如 `bg-primary`、`text-secondary`。

## 四、路由体系讲解

### 4.1 Next.js App Router 的工作方式

Next.js 13+ 使用**文件系统路由**，`src/app/` 下的目录结构就是网站的 URL 结构：

```
src/app/
├── layout.tsx              → 所有页面的根布局（包裹所有子页面）
├── page.tsx                → /
├── login/page.tsx          → /login
├── feedback/page.tsx       → /feedback
└── admin/feedbacks/page.tsx → /admin/feedbacks
```

加上 `basePath` 后，实际 URL 变为：

```
http://localhost:3000/tiantian-personal-website/
http://localhost:3000/tiantian-personal-website/login/
http://localhost:3000/tiantian-personal-website/feedback/
http://localhost:3000/tiantian-personal-website/admin/feedbacks/
```

### 4.2 页面与视图的分层设计

项目采用了"薄页面 + 胖视图"的模式：

```
page.tsx（薄页面）        →  views/Xxx.tsx（胖视图）
只做路由入口、Suspense 包裹      包含所有 UI 和业务逻辑
```

举例：`src/app/login/page.tsx` 只有 6 行：

```tsx
import { Suspense } from "react";
import Login from "@/views/Login";

export default function LoginPage() {
  return <Suspense><Login /></Suspense>;
}
```

这样做的好处：
- page.tsx 保持简洁，只负责路由级别的事（如 Suspense 边界、metadata 导出）
- 视图组件可以独立测试和复用
- Login 用了 `useSearchParams()`，Next.js 要求它必须包在 `<Suspense>` 里

### 4.3 路由守卫机制

管理端页面 `/admin/feedbacks` 需要登录才能访问，保护方式是用 `ProtectedRoute` 组件包裹：

```tsx
// src/app/admin/feedbacks/page.tsx
<ProtectedRoute>
  <AdminFeedbacks />
</ProtectedRoute>
```

`ProtectedRoute` 的工作流程：

```
                   ┌──── isReady? ────┐
                   │                  │
                   No                 Yes
                   │                  │
              显示"加载中…"      ┌─── token? ───┐
                               │              │
                              Yes              No
                               │              │
                          渲染子组件     router.replace(/login?from=当前路径)
```

`?from=` 参数用于登录成功后跳回原页面。

## 五、前后端数据交互逻辑

### 5.1 API 地址是怎么拼出来的

前端通过 `src/lib/api.ts` 的 `apiUrl()` 函数拼接 API 地址：

```
环境变量 NEXT_PUBLIC_API_BASE_URL = "http://localhost:3001"
调用 apiUrl("/api/auth/login")
结果 → "http://localhost:3001/api/auth/login"
```

如果不设置 `NEXT_PUBLIC_API_BASE_URL`，则使用相对路径 `/api/auth/login`（适用于前后端同域部署，由网关反代 `/api` 到后端）。

### 5.2 axios 拦截器自动处理 Token

`src/lib/axios.ts` 创建了一个带拦截器的 axios 实例 `apiClient`：

```
                          请求拦截器
                              │
              ┌───────────────┼───────────────┐
              │               │               │
       从 localStorage     如果 URL 不以     自动添加
       读取 access_token   http 开头，调用   Authorization: Bearer xxx
                           apiUrl() 补全      请求头

                          响应拦截器
                              │
                     ┌────── 401? ──────┐
                     │                  │
                    Yes                 No
                     │                  │
              清除 Token + 跳转      正常返回
              到 /login?from=当前路径
```

### 5.3 完整的登录流程

```
用户输入账号密码 → 点击"登录"
     │
     ▼
Login.tsx: onSubmit()
     │
     ├─ 调用 useAuth().login(username, password)
     │       │
     │       ▼
     │  AuthProvider: login()
     │       │
     │       ├─ POST /api/auth/login  { username, password }
     │       │       │
     │       │       ▼  (后端)
     │       │  AuthController.login()
     │       │       │
     │       │       ▼
     │       │  AuthService.login()
     │       │       │
     │       │       ├─ 1. 查数据库找 username
     │       │       ├─ 2. bcrypt.compare(密码, passwordHash)
     │       │       ├─ 3. jwt.signAsync({ sub: userId, username })
     │       │       └─ 返回 { success: true, access_token, token_type: "Bearer" }
     │       │
     │       ├─ 将 access_token 存入 localStorage
     │       ├─ 更新 React state (token, username)
     │       └─ 返回
     │
     └─ router.replace(from)  // 跳回之前想访问的页面
```

### 5.4 完整的反馈提交流程

```
已登录用户在 /feedback 页面填写表单 → 点击"提交反馈"
     │
     ▼
Feedback.tsx: onSubmit()
     │
     ├─ apiClient.post("/api/feedback", { name, email, content })
     │       │
     │       ├─ 请求拦截器自动添加 Authorization: Bearer <token>
     │       │
     │       ▼  (后端)
     │  FeedbackController.create()
     │       │
     │       ├─ @UseGuards(JwtAuthGuard) 验证 JWT
     │       │       │
     │       │       ├─ 从 Authorization Header 提取 Token
     │       │       ├─ JwtStrategy.validate() 解码并查数据库确认用户存在
     │       │       └─ 将 { userId, username } 注入 request.user
     │       │
     │       ├─ @Body() dto 经过 ValidationPipe 校验
     │       │       ├─ name: 非空, 最长 50 字符
     │       │       ├─ email: 合法邮箱格式
     │       │       └─ content: 非空, 最长 5000 字符
     │       │
     │       ├─ @CurrentUser() user 获取当前用户 ID
     │       │
     │       ▼
     │  FeedbackService.create(dto, userId)
     │       │
     │       ├─ 创建 Feedback 实体（name, email, content, userId）
     │       ├─ 存入 PostgreSQL feedbacks 表
     │       └─ 返回 { success: true, id: 新记录ID }
     │
     ├─ 显示"提交成功"提示
     └─ 3 秒后重置表单
```

### 5.5 管理端反馈列表的数据流

```
/admin/feedbacks 页面加载
     │
     ├─ ProtectedRoute 检查 token → 未登录跳 /login
     │
     ├─ AdminFeedbacks 组件挂载
     │       │
     │       ├─ useEffect → fetchFeedbacks()
     │       │       │
     │       │       ├─ apiClient.get("/api/feedback")
     │       │       │       │
     │       │       │       ▼  (后端)
     │       │       │  FeedbackController.findAll()
     │       │       │       │
     │       │       │       ▼
     │       │       │  FeedbackService.findAll()
     │       │       │       │
     │       │       │       ├─ 查询全部反馈，按 createdAt DESC 排序
     │       │       │       └─ 返回 { success: true, feedbacks: [...] }
     │       │       │
     │       │       └─ setFeedbacks(payload.feedbacks)
     │       │
     │       └─ setInterval(fetchFeedbacks, 3000)  // 每 3 秒轮询刷新
     │
     └─ 渲染卡片列表，显示 dayjs(...).fromNow() 相对时间
```

## 六、后端 API 路由总览

所有路由都有全局前缀 `/api`（在 `main.ts` 中设置 `app.setGlobalPrefix('api')`）：

| 方法 | 路径 | 是否需要 JWT | 说明 |
|------|------|:---:|------|
| GET | `/api/health` | 否 | 健康检查，返回 `{ ok: true, instanceId: "..." }` |
| POST | `/api/auth/login` | 否 | 登录，返回 JWT Token |
| GET | `/api/feedback` | 是 | 获取全部反馈列表 |
| POST | `/api/feedback` | 是 | 提交新反馈 |

## 七、数据库设计

### users 表

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | bigint | PK, 自增 | 用户 ID |
| `username` | text | UNIQUE, NOT NULL | 用户名 |
| `password_hash` | text | NOT NULL | bcrypt 哈希后的密码 |
| `role` | text | NOT NULL, 默认 `'user'` | 角色（`admin` / `user`） |
| `created_at` | timestamptz | 自动生成 | 创建时间 |

### feedbacks 表

| 列名 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | bigint | PK, 自增 | 反馈 ID |
| `name` | text | NOT NULL | 提交者姓名 |
| `email` | text | NOT NULL | 提交者邮箱 |
| `content` | text | NOT NULL | 反馈内容 |
| `user_id` | bigint | FK → users.id, ON DELETE SET NULL | 提交者用户 ID |
| `created_at` | timestamptz | 自动生成 | 创建时间 |

**关系**：`feedbacks.user_id → users.id`（多对一），删除用户时反馈的 `user_id` 置为 NULL 而不是级联删除。

## 八、启动前后端时发生了什么

### 8.1 后端启动流程（`pnpm dev:server`）

```
nest start --watch
     │
     ▼
main.ts bootstrap()
     │
     ├─ 1. loadEnvFiles()
     │       手动读取 backend/.env 和 ../.env，将键值对写入 process.env
     │       （只在 process.env 中不存在的 key 才写入，不覆盖已有的）
     │
     ├─ 2. fixLegacyUsersTableIfNeeded()
     │       用原始 pg Client 连接数据库，检查 users 表：
     │       - 如果表存在但缺少 username 列 → ALTER TABLE ADD COLUMN
     │       - 如果表存在但缺少 password_hash 列 → ALTER TABLE ADD COLUMN
     │       - 对 NULL 值的行回填默认值（username 填 "user_" + id，密码填占位哈希）
     │       目的：兼容旧版本数据库，让 TypeORM synchronize 不会因 NOT NULL 约束报错
     │
     ├─ 3. NestFactory.create(AppModule)
     │       │
     │       ├─ ConfigModule.forRoot() — 加载环境变量到 NestJS 配置服务
     │       │
     │       ├─ TypeORM 连接 PostgreSQL
     │       │       - 开发环境 synchronize: true → 自动建表/更新表结构
     │       │       - 生产环境 synchronize: false → 不自动修改表
     │       │
     │       ├─ AuthModule 初始化
     │       │       - 注册 JwtModule（密钥来自 resolveJwtSecret）
     │       │       - 注册 PassportModule + JwtStrategy
     │       │
     │       ├─ FeedbackModule 初始化
     │       │
     │       └─ SeedService.onModuleInit()  ← 自动执行
     │               │
     │               └─ AuthService.ensureDefaultAdmin()
     │                       - 读取 ADMIN_USERNAME（默认 "admin"）和 ADMIN_PASSWORD
     │                       - 开发环境未设密码则默认 "admin"
     │                       - 生产环境未设密码则跳过不创建
     │                       - 如果数据库中不存在该用户 → bcrypt 加密 → 插入
     │
     ├─ 4. app.setGlobalPrefix('api')
     │       所有路由自动加上 /api 前缀
     │
     ├─ 5. app.useGlobalPipes(ValidationPipe)
     │       - whitelist: true → 自动剥离 DTO 中未定义的字段
     │       - forbidNonWhitelisted: true → 传了多余字段直接报 400
     │       - transform: true → 自动类型转换
     │
     ├─ 6. app.enableCors(...)
     │       - 有 FRONTEND_ORIGIN 则只允许指定来源
     │       - 没有则 origin: true（反射任意来源，方便开发）
     │
     └─ 7. app.listen(port)
            默认 3001 端口启动
```

node scripts/dev.mjs

## 九、认证体系详解

### 9.1 JWT 的工作原理（在本项目中）

JWT（JSON Web Token）是一种无状态的认证方案。用户登录后，服务器签发一个加密 Token 给前端；此后前端每次请求带上这个 Token，服务器验证签名即可确认身份，无需查 Session。

```
登录成功后：
  后端签发 Token = jwt.sign({ sub: "1", username: "admin" }, 密钥, { expiresIn: "7d" })
  → eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxIi...

后续请求：
  前端带上 Header: Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
  → 后端验证签名 + 过期时间 → 解码出 { sub: "1", username: "admin" }
  → 查数据库确认用户存在 → 放行
```

### 9.2 密钥管理策略

`jwt-secret.util.ts` 的逻辑：

| 场景 | JWT_SECRET 环境变量 | 行为 |
|------|-------------------|------|
| 开发环境 | 未设置 | 使用硬编码的默认密钥 + 打印警告日志 |
| 开发环境 | 已设置 | 使用环境变量的值 |
| 生产环境 | 未设置 | **直接报错**，拒绝启动 |
| 生产环境 | 已设置 | 使用环境变量的值 |

### 9.3 默认管理员账号

| 环境 | 用户名 | 密码 | 行为 |
|------|--------|------|------|
| 开发环境（默认） | `admin` | `admin` | 首次启动自动创建 |
| 生产环境 | 由 `ADMIN_USERNAME` 指定 | 由 `ADMIN_PASSWORD` 指定 | 必须显式配置，否则不创建 |

## 十二、"后端重启清登录态"的演示机制

这是一个专为演示/开发设计的巧妙小功能：

1. 后端每次启动时生成一个随机 UUID 作为 `instanceId`
2. `GET /api/health` 返回这个 `instanceId`
3. 前端 `AuthProvider` 初始化时请求 `/api/health`，把 `instanceId` 存入 localStorage
4. 下次初始化时比较新旧 `instanceId`：
   - **不同** → 后端重启过 → 清除 Token → 用户需要重新登录
   - **相同** → 后端没重启 → 保留登录状态

**使用场景**：向老板演示时，每次重启 `pnpm dev:server` 都会自动回到登录页，方便展示完整的登录流程。

## 十三、旧数据库兼容机制

`fix-legacy-users-table.ts` 解决的是数据库表结构迁移的问题。

**背景**：如果数据库中已经存在一个旧版本的 `users` 表（比如之前用的字段名不同），TypeORM 的 `synchronize: true` 会尝试加上 `username` 和 `password_hash` 列并设为 `NOT NULL`，但旧数据行这些列是 NULL，会导致约束冲突报错。

**解决策略**：在 TypeORM 初始化之前（NestFactory.create 之前），用原始 SQL 先做修复：

1. 检查 `users` 表是否存在
2. 如果缺少 `username` 列 → `ALTER TABLE ADD COLUMN`
3. 将所有 `username` 为 NULL 的行填充为 `"user_" + id`
4. 如果缺少 `password_hash` 列 → `ALTER TABLE ADD COLUMN`
5. 将所有 `password_hash` 为 NULL 的行填充占位哈希

之后 TypeORM 的 `synchronize` 再将列改为 `NOT NULL`、加 `UNIQUE` 约束时就不会出错了。
