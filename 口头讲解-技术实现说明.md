# 口头讲解用：技术实现说明

> 根据当前仓库实现整理（前端 Vite + React，后端 NestJS + PostgreSQL）。若与 `CODE_GUIDE.md` 中旧版描述不一致，以本文件为准。

---

## 一、整体架构（前后端怎么连、数据怎么走）

**通俗版：**  
浏览器里跑的是 **React 单页应用**（页面切换不整页刷新）。需要数据时，前端用 **HTTP 请求**访问后端的 **REST 风格接口**（路径都以 `/api` 开头）。后端用 **NestJS** 处理请求、查 **PostgreSQL**、再返回 JSON。

**开发时：**  
Vite 把以 `/api` 开头的请求**代理**到本机 `localhost:3000`，这样前端写 `/api/...` 不用操心跨域。  
**上线时：** 可通过环境变量 `VITE_API_BASE_URL` 指到真实后端完整地址。

**数据流（可画成一条线）：**  
用户操作 → React 页面 → `axios`（`apiClient`）→ `/api/...` → Nest 控制器 → Service → TypeORM → 数据库 → 原路返回 JSON → 页面更新界面。

**你可以这样讲：**  
「前端是 React，用 axios 调后端；后端是 Nest，统一前缀 `/api`，数据存在 PostgreSQL。开发时用 Vite 代理把 `/api` 转到后端，部署时用环境变量配后端地址。」

### 【需提前搞懂】

- **单页应用 + 路由**：URL 变、组件换，不是传统多页跳转。
- **代理**：开发时浏览器只认前端端口，由 Vite 转发 `/api` 到后端。
- **JSON**：前后端约定好字段，用请求/响应传数据。

## 二、核心模块与实现步骤

### 1. 路由与页面

- `BrowserRouter` + `basename`：站点部署在子路径 `/tiantian-personal-website/` 下，路由要匹配。
- 首页 `/`、登录 `/login`、反馈 `/feedback`、管理页 `/admin/feedbacks`（管理页包在 **ProtectedRoute** 里）。

**你可以这样讲：**  
「路由决定哪个地址显示哪个页面；管理页外面套了一层保护，没登录会先被拦到登录页。」

### 【需提前搞懂】

- `basename` 和部署路径的关系（否则线上路径会错）。

---

### 2. 登录流程（端到端）

1. 用户在登录页提交账号密码（`react-hook-form` 做校验）。
2. 前端 `POST /api/auth/login`， body 里用户名、密码。
3. 后端 `AuthService`：按用户名查库 → **bcrypt** 比对密码哈希 → 通过则用 **JWT** 签发 `access_token`。
4. 前端把 token 和用户名存 **localStorage**，并更新 **AuthProvider** 里的状态。
5. 之后带鉴权的请求通过 axios **请求拦截器**自动加头：`Authorization: Bearer <token>`。

**你可以这样讲：**  
「密码不会在库里明文保存，而是哈希比对；登录成功发一个 JWT，前端存起来，以后每次请求自动带上，后端验证通过才放行。」

### 【需提前搞懂】

- **JWT 是什么**：一段签过名的字符串，里面带有用户标识，用来证明「我是谁」，不是把密码每次传来传去。
- **localStorage**：浏览器本地存储，刷新页面还在（个人站/演示场景；生产环境还需考虑 XSS 等，不必对老板展开太深）。

---

### 3. 受保护页面（管理端）

- `ProtectedRoute`：先看 **AuthProvider 是否就绪**（`isReady`），再等 **token**。没有 token 就 **Navigate** 到 `/login`，并把「从哪来」记在 `location.state`，登录成功可以跳回去。
- axios **响应拦截器**：若返回 **401**，清 token、跳转登录页（并带 `from` 查询参数），避免「过期了还假登录」。

**你可以这样讲：**  
「管理页不是谁都能进：组件先判断有没有 token；接口如果返回未授权，也会统一踢回登录。」

### 【需提前搞懂】

- **401**：未登录或 token 无效时后端的典型状态码。

---

### 4. 反馈：提交与列表

- **提交反馈** `POST /api/feedback`：控制器上用了 **JwtAuthGuard**，必须登录；后端把当前用户 id 写入反馈记录（`FeedbackService.create`）。
- **列表** `GET /api/feedback`：当前实现是**公开**的，返回全部反馈；管理页用 `apiClient.get` 拉取，并做了定时刷新（例如每 3 秒）方便演示「新反馈出现」。

**你可以这样讲：**  
「写反馈要登录，这样每条能关联到账号；列表页拉全量数据展示，我这边还做了定时刷新，演示时比较直观。」

### 【需提前搞懂】

- **GET 与 POST 分工**：一个读、一个写；本项目里「写」要登录，「读」目前是开放的——若被问到安全，可说明：个人站演示可以这样；若要对列表也保密，可以给 GET 加守卫或分页鉴权（可扩展点）。

---

### 5. 首页等内容页

- `Home.tsx` 里大量是个人介绍、项目、时间线等，多为**前端写死或本地数据结构**，主要不经过后端（和反馈/登录那条线分开）。

**你可以这样讲：**  
「展示型内容主要在前端维护，和业务接口分开，改文案不用动服务器。」

## 三、关键决策（为什么这样选）

| 决策 | 可以这样解释（口语） |
|------|----------------------|
| React + Vite | 生态成熟，开发热更新快，构建简单。 |
| TypeScript | 前后端都有类型，改接口不容易写错字段。 |
| NestJS | 后端结构清晰（模块、控制器、服务），适合 REST + 鉴权。 |
| PostgreSQL + TypeORM | 关系型数据、反馈要持久化；ORM 减少手写 SQL 的量。 |
| JWT + Bearer | 无状态登录，后端不用为每个请求查 session 表（对新手解释到这就够）。 |
| axios 拦截器 | 登录态加头、401 统一处理，避免每个页面复制粘贴。 |

**你可以这样讲：**  
「我选的是常见、文档多的组合：React 做界面，Nest 做 API，Postgres 存数据；登录用 JWT，用 axios 拦截器统一带 token 和处理过期。」

### 【需提前搞懂】

- 能说出 **「为什么用 JWT」** 一句话版本即可（见上表）。
- **TypeORM 的 synchronize**：非生产环境可能自动同步表结构，生产一般要迁移策略——若被问到，可说「小项目开发期图方便，上线会再收紧」。

## 四、典型问题与解决思路（含项目里真实细节）

1. **跨域（开发）**  
   - 现象：前端端口和后端端口不同，浏览器会拦请求。  
   - 做法：Vite `proxy` 把 `/api` 转到后端，同源策略上就像只访问前端。

2. **部署在子目录**  
   - 现象：资源路径、路由全错。  
   - 做法：`vite.config` 里设 `base`，`BrowserRouter` 设同样 `basename`。

3. **token 过期或未授权**  
   - 现象：接口 401。  
   - 做法：响应拦截器清本地登录信息并跳转登录页。

4. **开发时后端重启还想「重新演示登录」**（代码中已实现）  
   - `AppController` 的 `health` 返回 **`instanceId`**；`AuthProvider` 启动时请求 health，若 id 变了就认为后端重启过，清掉本地 token，方便每次重启后端都从头登录演示。

**你可以这样讲：**  
「跨域我用开发代理解决；子目录部署要配 base；401 统一拦截；另外我加了一个小检测，后端重启就清登录，方便演示。」

### 【需提前搞懂】

- **CORS / 代理** 二选一能说清一句：开发用代理，生产可配 `FRONTEND_ORIGIN` 等（见 `backend/src/main.ts` 中的 CORS 配置）。

### 2. axios 自动带 token + 401 处理

文件：`src/lib/axios.ts`

```ts
export const apiClient = axios.create();

apiClient.interceptors.request.use((config) => {
  const token = localStorage.getItem(TOKEN_KEY);
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  const u = config.url;
  if (u && !u.startsWith("http")) {
    config.url = apiUrl(u);
  }
  return config;
});

apiClient.interceptors.response.use(
  (res) => res,
  (err: unknown) => {
    const status = axios.isAxiosError(err) ? err.response?.status : undefined;
    if (status === 401) {
      localStorage.removeItem(TOKEN_KEY);
      localStorage.removeItem("username");
      const path = window.location.pathname;
      const base = import.meta.env.BASE_URL.replace(/\/$/, "") || "";
      if (!path.endsWith("/login")) {
        const qs = `?from=${encodeURIComponent(path + window.location.search)}`;
        window.location.href = `${base}/login${qs}`;
      }
    }
    return Promise.reject(err);
  },
);
```

### 3. 后端登录核心（查库 → 比对哈希 → 签发 JWT）

文件：`backend/src/auth/auth.service.ts`

```ts
async login(dto: LoginDto) {
  const user = await this.users.findOne({
    where: { username: dto.username.trim() },
  });
  if (!user) {
    throw new UnauthorizedException('Invalid credentials');
  }
  const ok = await bcrypt.compare(dto.password, user.passwordHash);
  if (!ok) {
    throw new UnauthorizedException('Invalid credentials');
  }
  const access_token = await this.jwt.signAsync({
    sub: user.id,
    username: user.username,
  });
  return { success: true as const, access_token, token_type: 'Bearer' as const };
}
```

## 附录：可能被追问的简短答法（Q&A 小抄）

| 可能的问题 | 可以这样答（新手友好） |
|------------|-------------------------|
| JWT 存在 localStorage 安全吗？ | 个人项目演示够用；正式产品还会配合 HTTPS、防 XSS、以及更细的策略（httpOnly Cookie 等），可以按阶段升级。 |
| 为什么反馈列表 GET 不鉴权？ | 当前是演示向：管理页主要给自己看；若要对外保密，可以给列表接口也加登录或权限。 |
| 前后端怎么部署？ | 前端构建静态文件；后端单独跑 Nest，配好 `DATABASE_URL` 和 `VITE_API_BASE_URL`（或同源反代）即可。 |
| 为什么用 bcrypt？ | 密码不能明文存库；bcrypt 是常用的单向哈希，用来比对「用户输入是否和当初存的一致」。 |
